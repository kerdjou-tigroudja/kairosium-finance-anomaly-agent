# ADR-005 — CI/CD Cloud Build (statut et usage)

**Statut :** Accepté (cas d’étude T2) — fichier dédié pour satisfaire la traçabilité Constitution §7.4 / Grille v2.3.

## Contexte

La Constitution v2.3 §7.4 décrit une **CI bloquante** (Cloud Build sur merge) avec export des résultats vers BigQuery `test_runs`. Sur un dépôt démo, l’historique a été de reporter l’automatisation complète tout en conservant **pytest** et `scripts/import_test_results.py` comme source de vérité fonctionnelle.

## Décision

1. **`cloudbuild.yaml`** à la racine du paquet `kairosium-finance-anomaly-agent` exécute `uv sync` + `pytest` sur chaque build déclenché (même logique que `make test` sans l’import BQ conditionnel, sauf variables projet fournies au build).
2. La **décision de différé** (pas de trigger obligatoire sur ce repo demo) reste documentée ici : la **lettre** Cloud Build merge-blocking n’est remplie que si un administrateur GCP attache ce fichier à un trigger sur la branche protégée.
3. **Critère de réactivation stricte :** projet client avec trigger Cloud Build sur `main` / `production`, secrets ADC ou workload identity, et optionnellement étape d’import JUnit → `test_runs`.

## Conséquences

- Les revues humaines restent le garde-fou si aucun trigger n’est configuré.
- Dès qu’un trigger pointe vers ce `cloudbuild.yaml`, l’alignement avec la Constitution §7.4 est **opérationnel** sans changer le texte de l’ADR.
