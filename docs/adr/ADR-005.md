# ADR-005 — CI / Cloud Build (niveau d’automatisation retenu)

**Statut :** Accepté

## Contexte

Un pipeline CI entièrement bloquant sur merge (Cloud Build, résultats exportables, gouvernance stricte) est souhaitable pour un produit en croissance, mais n’est pas le même investissement qu’un **pilote mono-environnement** avec une petite équipe.

## Décision

1. Le dépôt fournit un **`cloudbuild.yaml`** exécutant `uv sync` et `pytest` quand un build est déclenché (même logique que `make test`, sans l’import BigQuery des résultats de tests si les variables de projet ne sont pas fournies).
2. **Aucun trigger n’est imposé par le code** : tant qu’un administrateur GCP n’attache pas ce fichier à un trigger sur une branche protégée, la **CI n’est pas bloquante** sur le merge. C’est un choix assumé pour limiter l’**overhead d’infrastructure** pendant la phase pilote.
3. En attendant, la **vérification locale** (`Makefile`, tests) reste le garde-fou principal et suffit à ce stade.
4. **Critère de passage à une CI bloquante** : exigence projet (trigger Cloud Build sur `main` / `production`, secrets via ADC ou Workload Identity, éventuellement import JUnit / traçabilité des runs de tests en base).

## Conséquences

- La revue humaine et les exécutions locales de tests suppléent l’absence de trigger.
- Dès qu’un trigger pointe vers `cloudbuild.yaml`, le pipeline automatisé est opérationnel **sans** modifier le texte de l’ADR.

## Références

- `Makefile`, cible `test` (pytest, import optionnel des résultats vers BigQuery si `GOOGLE_CLOUD_PROJECT` est défini).
- Fichier `cloudbuild.yaml` à la racine du paquet applicatif.
